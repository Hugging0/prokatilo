# 🧠 ПРОКАТило — Мастер-контекст и Правила ИИ (AI_RULES.md)

## 1. Контекст проекта
- **Название:** ПРОКАТило.
- **Суть:** MVP сервиса аренды вещей и техники. 
- **Ключевое УТП:** Доставка за 15 минут, аренда без залога.
- **Бизнес-модель:** B2C (вещи сдаются централизованно владельцем сервиса, это НЕ шеринг между соседями).
- **Стартовый инвентарь:** PlayStation, PlayStation VR, робот-мойщик окон, моющий пылесос для бытовой мебели.
- **Архитектура:** Раздельный Fullstack (Next.js на фронтенде, FastAPI на бэкенде).

## 2. Технологический стек
- **Frontend:** Next.js (App Router), React, Tailwind CSS, TypeScript, Zustand (стейт), Lucide React (иконки).
- **Backend:** FastAPI, Python, SQLAlchemy, Pydantic.

## 3. Железные правила написания кода (НЕ НАРУШАТЬ)
1. **Строгий TypeScript:** Никаких `any`. Все интерфейсы (`Item`, `Order`, `User`) должны быть строго описаны в `src/types`.
2. **Язык интерфейса:** Весь UI, плейсхолдеры и уведомления — СТРОГО на русском языке.
3. **Модульность (FSD-подобная архитектура):**
   - Страницы лежат только в `src/app`.
   - Переиспользуемые кнопки, инпуты — в `src/components/ui`.
   - Сложные блоки (список заказов, форма) — в `src/components/features`.
   - Утилиты и константы — в `src/lib`.
4. **Запрет на спагетти-код:** Компоненты не должны превышать 150-200 строк. Если файл растет — декомпозируем.
5. **Чистый импорт:** Использовать алиасы `@/components/...` вместо относительных путей `../../`.

## 4. Правила рабочего процесса (Workflow)
- Каждый логический шаг завершается коммитом.
- Использовать конвенцию коммитов (feat: ..., fix: ..., chore: ..., refactor: ...).
- При добавлении новых сущностей (например, новой таблицы в БД или нового глобального стейта) этот файл ДОЛЖЕН БЫТЬ ОБНОВЛЕН.

## 5. Backend/API правила

1. Backend запускается как Python package через `uvicorn app.main:app`.
2. PostgreSQL подключается через async SQLAlchemy engine и async-драйвер `asyncpg`.
3. При использовании Supabase pooler с asyncpg отключается prepared statement cache через `statement_cache_size=0`.
4. Статусы заказов задаются через единый enum `OrderStatus`.
5. Тарифы аренды задаются через единый enum `TariffType`.
6. `Base.metadata.create_all` разрешён только для раннего MVP/dev-режима. Перед production-деплоем нужен Alembic.

## 6. Frontend domain rules

1. Backend DTO-типы и UI-типы не смешиваются напрямую.
2. Статусы заказов и тарифы хранятся в `src/lib`, а не внутри страниц.
3. Mock-данные хранятся в `src/lib/mock-data.ts`.
4. `src/app/page.tsx` может временно быть orchestration-файлом, но бизнес-константы и доменные типы должны быть вынесены.
5. Стартовый mock-каталог должен соответствовать B2C-модели ПРОКАТило и стартовому инвентарю из мастер-контекста.

## 7. Brand/UI правила

1. Основной визуальный стиль ПРОКАТило — mobile-first, яркий, дружелюбный, с акцентом на оранжево-розовый градиент.
2. Основной бренд-градиент: `from-amber-400 via-orange-500 to-rose-600`.
3. Логотип в UI пишется как `ПРОКАТило` и визуально оформляется жирным курсивом: `font-black italic tracking-tighter`.
4. Главный экран и auth-экран должны использовать брендовый градиент, а не полностью чёрный header.
5. Чёрный/slate используется для контраста и CTA, но не должен заменять основной брендовый оранжево-розовый стиль.

## 8. PWA правила

1. PWA metadata задаётся через `src/app/layout.tsx` и `src/app/manifest.ts`.
2. Название приложения в metadata и manifest берётся из `src/lib/brand.ts`.
3. Основной `theme_color` должен соответствовать бренду ПРОКАТило, сейчас `#f97316`.
4. Перед production-деплоем временная SVG-иконка должна быть заменена на production PNG-иконки `192x192`, `512x512` и maskable-иконку.

## 9. Product copy rules

1. В пользовательском UI запрещены технические значения вроде `3h`, `6h`, `24h`; они форматируются как `3 часа`, `6 часов`, `сутки` или `3 ч`, `6 ч`, `24 ч`.
2. UI не должен использовать peer-to-peer формулировки: “владелец”, “сосед”, “арендодатель”, если речь о клиентском сценарии.
3. Бизнес-модель в текстах — B2C: “сервис”, “оператор”, “курьер”, “поддержка”.
4. Статусы заказа должны иметь клиентское объяснение следующего шага.
5. В MVP нельзя создавать впечатление реального списания денег, если платёж ещё не подключён.
6. Все пользовательские тексты, плейсхолдеры, toast-уведомления и подписи кнопок должны быть на русском языке и храниться в `src/lib/copy.ts`, если они используются в нескольких местах или отражают бизнес-логику.

## 10. Frontend component architecture rules

1. `src/app/page.tsx` должен быть orchestration-файлом: state, handlers, выбор текущего экрана.
2. Feature JSX-экраны должны жить в `src/components/features/*`.
3. Нижняя навигация живёт в `src/components/layout/AppNavigation.tsx`.
4. Toast/переиспользуемые небольшие UI-элементы живут в `src/components/ui`.
5. При декомпозиции запрещено менять брендовый стиль или продуктовые тексты без отдельного решения.
6. Каждый feature-компонент желательно держать до 200 строк; если компонент растёт, его нужно дробить дальше.

## 11. Frontend API integration rules

1. Frontend ходит к backend через `src/lib/api/*`, а не напрямую из компонентов.
2. `NEXT_PUBLIC_API_URL` задаёт backend base URL; локальный fallback — `http://localhost:8000`.
3. Backend DTO-типы не используются напрямую в UI-компонентах; они преобразуются через mapper в `AppItem`, `AppOrder` и другие UI-модели.
4. Пока backend API может быть недоступен, каталог обязан иметь fallback на `src/lib/mock-data.ts`.
5. Компоненты не должны знать о структуре backend DTO.
