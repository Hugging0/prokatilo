# Refactoring rules

## Цель

Рефакторинг должен улучшать понятность кода без изменения поведения приложения.

## Правила

1. Сначала вынести чистые функции.
2. Потом покрыть imports.
3. Потом вынести UI-компоненты.
4. Потом вынести hooks.
5. После каждого этапа проверять lint/build.
6. Не менять тексты и UI одновременно с логическим рефакторингом, если это не требуется.
7. Не менять API payloads.
8. Не менять типы backend DTO без backend-задачи.
9. Не удалять сценарий, если он кажется лишним, без проверки использования.
10. Если действие пока заглушка, оставить честный disabled/toast, но не делать вид, что оно работает.

## Definition of safe refactoring

- Визуально приложение работает так же.
- Пользовательские сценарии не сломаны.
- Imports понятные.
- Файлы стали меньше.
- Повторяющаяся логика вынесена.
- `npm run lint` проходит.
